23.1 Introducción

Los sistemas de archivos y los datos que contienen son componentes extremadamente importantes de cualquier sistema, por lo que saber cómo solucionar problemas cuando ocurren es igualmente crítico. Esta es una de las principales razones por las que la información que se almacena en un sistema de archivos debe copiarse en múltiples ubicaciones o asegurarse mediante copias de seguridad más tradicionales. También destaca lo importante que es entender cómo reparar un sistema de archivos, en caso de que se rompa.

Los sistemas de archivos pueden requerir reparación por varias razones:

  • Corte repentino de energía que resulta en un desmontaje incorrecto de las particiones. Típicamente se utiliza una fuente de alimentación ininterrumpida (UPS) para evitar esto.
  • Expulsar medios extraíbles de lectura/escritura, como una memoria USB o una unidad flash, antes de que se haya producido un desmontaje limpio (es decir, retirar el medio antes de emitir el comando umount).
  • Pérdida de conectividad de red entre un host y una partición montada como sistema de archivos de red (NFS).
  • Desgaste normal que lleva a sectores de disco corruptos y fallas graduales del disco.
  • Sobretensiones eléctricas o exposición a campos magnéticos.
  • Malware.

23.2 El comando fsck

El comando fsck se utiliza para verificar sistemas de archivos en busca de problemas de consistencia y para reparar esos problemas cuando se encuentran. El comando fsck tiene algunas similitudes con el comando mkfs. En primer lugar, es una herramienta genérica para usar con sistemas de archivos; su propósito es verificar los sistemas de archivos en busca de errores e intentar repararlos. En segundo lugar, fsck es en realidad una interfaz para los diversos verificadores de sistemas de archivos, lo que significa que llamará al comando de backend apropiado, como el comando e2fsck, basándose en el tipo de sistema de archivos.

Los archivos que vinculan la interfaz con el backend se nombran siguiendo patrones como fsck.TIPO-DE-SISTEMA-DE-ARCHIVOS, por ejemplo fsck.ext4 o fsck.vfat. Estos archivos entre los comandos de interfaz y backend pueden ser una excelente manera de encontrar documentación.

Por ejemplo, para descubrir el comando real específico del sistema de archivos que se utiliza para verificar sistemas de archivos vfat, ejecute el comando man fsck.vfat. Esto mostrará la página del manual para el comando de backend real para verificar sistemas de archivos vfat, que es el comando dosfsck. Del mismo modo, ver la página del manual de fsck.ext4 muestra el comando e2fsck utilizado para verificar sistemas de archivos ext2, ext3 y ext4.

No es necesario saber qué backend usar. Simplemente ejecute el comando fsck con la ruta del sistema de archivos o su punto de montaje. El comando fsck determinará el programa específico del sistema de archivos a llamar basándose en el tipo de sistema de archivos de la entrada en el archivo /etc/fstab. Si el sistema de archivos no tiene una entrada en el archivo /etc/fstab, entonces se puede usar la opción -t con el comando fsck para especificar el tipo de sistema de archivos.

Advertencias sobre el comando fsck

Aunque esta herramienta está diseñada para verificar y reparar sistemas de archivos, ¡también puede dañar sistemas de archivos! La utilidad fsck no está diseñada para ejecutarse en sistemas de archivos actualmente en uso (sistemas que ya están montados). Si el comando fsck se ejecuta en un sistema de archivos actualmente montado, aparecerá el siguiente error:

Nota

El siguiente ejemplo puede no coincidir con la salida en nuestro entorno virtual.

root@localhost:~# fsck /dev/sda1
fsck de util-linux-ng 2.17.2

e2fsck 1.41.12 (17-May-2010)
/dev/sda1 está montado.

¡¡ADVERTENCIA!! El sistema de archivos está montado. Si continúa, ***CAUSARÁ*** un daño ***GRAVE*** al sistema de archivos.

¿Realmente desea continuar (s/n)? no

verificación abortada

En este caso, definitivamente escriba n para no. Tenga en cuenta que, si el sistema de archivos ya está montado, no necesita ser verificado. El hecho mismo de que esté montado significa que está funcionando correctamente.

Se debe implementar una política de respaldo para facilitar la recuperación de archivos importantes en caso de que se pierdan o se corrompan. Es muy común que no se consideren los respaldos hasta que sea demasiado tarde.

Considera esto

Cuando se trata de respaldos, tarde puede ser mejor que nunca. Si un sistema de archivos está dañado y no se puede montar, considere hacer un respaldo con el comando dd antes de intentar repararlo con el comando fsck. De esa manera, si la reparación que realiza el comando fsck empeora las cosas de lo que estaban originalmente, aún es posible volver a la condición preservada en el respaldo.

Una de las opciones más útiles para el comando fsck es la opción -y, que hará que fsck asuma una respuesta afirmativa a todas las preguntas que normalmente haría. El uso de esta opción puede evitar horas escribiendo la letra y en respuesta a preguntas de la herramienta como, ¿Le gustaría arreglar esto?

Dado que el sistema de archivos raíz no se puede desmontar mientras el sistema está en ejecución, puede ser difícil verificarlo. Un enfoque para verificar el sistema de archivos raíz es iniciar otro sistema operativo Linux con un sistema de archivos raíz diferente. Un enfoque común es descargar y usar un Live CD-ROM, DVD o USB, que son sistemas operativos Linux de arranque que caben en esos medios.

Otro enfoque no interactivo es forzar la verificación de un sistema de archivos, ejecutando el comando touch /forcefsck como usuario root. Si este archivo existe en el momento del arranque, el comando fsck se ejecuta en todos los sistemas de archivos en el archivo /etc/fstab que tienen un valor distinto de cero para la columna FSCK. Una vez completado, el archivo /forcefsck se eliminará y la marca de tiempo del archivo /.autofsck se actualizará a la hora en que finalizó la verificación.

Para particiones que se pueden desmontar mientras el sistema está en ejecución, el comando fsck se puede iniciar interactivamente para realizar la verificación. Por ejemplo:

Nota

Los siguientes ejemplos pueden no coincidir con la salida en nuestro entorno virtual.

root@localhost:~# mount | grep /data
/dev/sdb1 on /data type ext4 (rw)
root@localhost:~# umount /data
root@localhost:~# fsck /dev/sdb1
fsck de util-linux-ng 2.17.2
e2fsck 1.41.12 (17-May-2010)
/dev/sdb1: limpio, 11/12824 archivos, 6532/51200 bloques

Tenga en cuenta que el comando fsck no realizó una verificación del sistema de archivos en el ejemplo anterior. Dado que el sistema de archivos se desmontó limpiamente (lo que también ocurriría durante un apagado o reinicio normal del sistema), el estado del sistema de archivos se estableció en limpio. El comando fsck no realiza ninguna verificación cuando el estado del sistema de archivos es limpio.

Se puede forzar una verificación del sistema de archivos usando la opción -f:

root@localhost:~# fsck -f /dev/sdb1
fsck de util-linux-ng 2.17.2
e2fsck 1.41.12 (17-May-2010)
Paso 1: Verificando inodos, bloques y tamaños
Paso 2: Verificando la estructura del directorio
Paso 3: Verificando la conectividad del directorio
Paso 4: Verificando los recuentos de referencias
Paso 5: Verificando la información de resumen del grupo
/dev/sdb1: 11/12824 archivos (9.1% no contiguos), 6532/51200 bloques

Cuando una partición tiene un problema, intentar montarla puede resultar en un mensaje de error como este:

mount: debe especificar el tipo de sistema de archivos

Normalmente, el comando mount leerá el primer bloque del sistema de archivos (el superbloque) donde se guarda la información sobre el tipo de sistema de archivos. Cuando un superbloque está dañado, el comando mount no puede leer esa información.

En este caso, use el comando fsck para reparar el sistema de archivos. La salida puede verse algo así:

fsck de util-linux-ng 2.17.2

e2fsck 1.41.12 (17-May-2010)

fsck.ext2: Superbloque inválido, intentando bloques de respaldo...
mydata no se desmontó limpiamente, se forzó la verificación.
Paso 1: Verificando inodos, bloques y tamaños
Paso 2: Verificando la estructura del directorio
Paso 3: Verificando la conectividad del directorio
Paso 4: Verificando los recuentos de referencias
Paso 5: Verificando la información de resumen del grupo

mydata: ***** EL SISTEMA DE ARCHIVOS FUE MODIFICADO *****
mydata: 11/25688 archivos (0.0% no contiguos), 4800/102400 bloques

Observe que el texto resaltado de la salida muestra que el comando intentó usar bloques de respaldo. El comando fsck a menudo podrá encontrar una copia de seguridad del superbloque que se creó la última vez que se modificó el sistema de archivos.

23.3 El comando e2fsck

El comando e2fsck es el verificador de sistemas de archivos llamado por fsck para sistemas de archivos ext2, ext3 y ext4. Existen opciones que se pueden usar con el comando e2fsck que no están documentadas en la página del manual del comando fsck. Estas opciones específicas del sistema de archivos se pueden usar directamente con el comando e2fsck o con el comando fsck (que pasará las opciones al comando e2fsck).

Por ejemplo, el comando fsck puede requerir la ubicación exacta de un superbloque de respaldo. El superbloque es un componente crítico del sistema de archivos y, cuando está corrupto, es difícil de recuperar. Los superbloques de respaldo son utilizados por el comando fsck para reparar un superbloque corrupto.

Si se utilizaron configuraciones personalizadas al crear un sistema de archivos, los superbloques de respaldo pueden no estar en una ubicación estándar. En ese caso, la salida del comando mkfs debe documentarse porque esa salida incluye la ubicación de los superbloques de respaldo. Estos superbloques de respaldo se pueden localizar utilizando el comando dumpe2fs:

Nota

Los siguientes ejemplos pueden no coincidir con la salida en nuestro entorno virtual.

root@localhost:~# dumpe2fs /dev/sdb1 | grep superblock
dumpe2fs 1.41.12 (17-May-2010)
  Superbloque primario en 1, Descriptores de grupo en 2-2
  Superbloque de respaldo en 8193, Descriptores de grupo en 8194-8194
  Superbloque de respaldo en 24577, Descriptores de grupo en 24578-24578
  Superbloque de respaldo en 40961, Descriptores de grupo en 40962-40962

Use la opción -b con el comando e2fsck para especificar un respaldo cuando el superbloque primario se haya corrompido. Por ejemplo:

root@localhost:~# e2fsck -b 8193 /dev/sdb1

El comando fsck también podría usarse con la misma opción, ya que pasará esta opción al comando e2fsck:

root@localhost:~# fsck -b 8193 /dev/sdb1

Existen otras opciones para automatizar el comando e2fsck. La opción -n asume que la entrada para cada solicitud es "no". La opción -n puede ser útil al ejecutar el comando para probar posibles problemas, pero optando por no repararlos en este momento. La salida se puede capturar para un análisis y discusión posterior, quizás con otro administrador.

23.4 El directorio lost+found

En la mayoría de los casos, después de ejecutar el comando fsck, el sistema de archivos reparado se monta y no se requiere ninguna acción adicional. Sin embargo, si el comando fsck produce errores de archivos no referenciados, puede ser prudente buscar dentro del directorio lost+found que se encuentra en el punto de montaje del sistema de archivos. Esta es la ubicación donde se almacenan todos los archivos perdidos.

Un archivo puede estar "perdido" si su nombre de archivo se corrompe y, por lo tanto, no se puede colocar en un directorio normal. Después de fsck, dicho archivo se reubica en el directorio lost+found, y su número de inodo aparecerá en lugar del nombre del archivo.

Nota

Los siguientes ejemplos pueden no coincidir con la salida en nuestro entorno virtual.

root@localhost:~# ls -l /data
lost+found
root@localhost:~# ls -l /data/lost+found
total 5
-rw-r--r--. 1 root root   83 Mar  4 06:37 #34234
-rw-r--r--. 1 root root 1911 May 20  2013 #4596

¿Qué se debe hacer con estos "pobres archivos perdidos"? Sin un nombre, puede ser difícil encontrar a dónde pertenecen, pero tenga en cuenta que el comando file se puede usar para determinar el tipo de archivo. Para archivos de texto, ver el contenido puede ser útil. Las marcas de tiempo y la propiedad de los archivos también pueden ayudar a determinar a dónde pertenecen estos archivos.

23.5 Reparación de sistemas de archivos XFS

El sistema de archivos XFS tiene un amplio uso. Algunas distribuciones lo utilizan para la mayoría, si no todos, los sistemas de archivos, otras lo utilizan como sistema de archivos para la parte del directorio personal del usuario de un sistema. Es estable, con muchas funciones y de confianza.

Cada sistema de archivos, por muy fiable que sea, necesita ser reparado de vez en cuando, ya que ocurren cortes de energía, la corrupción del sistema de archivos ocurre debido a programas maliciosos, etc.

Los sistemas de archivos XFS se reparan de manera diferente a otros sistemas de archivos. Como XFS es capaz de contener un número inmensamente grande de archivos (inodos), esto haría que la verificación en el momento del arranque fuera poco práctica, incluso usando un diario del sistema de archivos, lo cual XFS hace.

XFS se basa en el diario para la mayoría de las correcciones de errores, o cuando se trata de que el sistema de archivos se marque como no haber sido desmontado correctamente. Si un sistema de archivos no se desmonta limpiamente, se marca como que necesita ser verificado, lo que XFS hace reproduciendo el registro del diario; generalmente esto es suficiente para asegurar que los datos se escriban correctamente.

En los casos en que los metadatos o la estructura de XFS se han corrompido, puede ocurrir un error como mount: Structure needs cleaning, y la utilidad xfs_repair deberá ejecutarse en el sistema de archivos para reparar cualquier inconsistencia.

Considera esto

Siempre que un sistema de archivos parece estar corrupto, es mejor tener una copia de seguridad de los datos en el sistema de archivos. Los comandos xfs_copy y xfsdump pueden ser útiles en ese sentido. El comando xfs_copy puede crear una duplicación exacta del sistema de archivos cuestionable para experimentar con las reparaciones.

Estos comandos están fuera del alcance de los objetivos y de este curso.

En el caso de que deba intentar reparar un sistema de archivos XFS, es bastante sencillo de usar. El comando xfs_repair solo se puede ejecutar en un sistema de archivos desmontado, lo que significa que si el sistema de archivos XFS en un dispositivo se usa como directorio /home, entonces todos los usuarios deben estar fuera del sistema de archivos para poder desmontarlo y repararlo.

Para desmontar el sistema de archivos /dev/sda3, como usuario root (y no en la estructura del directorio /home) use el comando:

Nota

Los siguientes ejemplos pueden no coincidir con la salida en nuestro entorno virtual.

root@localhost:~# umount /dev/sda3

Para usar xfs_repair en el sistema de archivos, use el siguiente comando:

    	- agno = 1

Nota

En ciertas circunstancias, puede ser ventajoso borrar el registro del diario del sistema de archivos XFS, como cuando el registro del diario se ha corrompido. Borrar un registro del diario XFS es una medida extrema y solo debe hacerse como último recurso.

Para borrar el registro del diario de un sistema de archivos XFS, use el comando:

root@localhost:~# xfs_repair -L /dev/sda3

En situaciones en las que se desea una reparación manual del sistema de archivos XFS, el comando xfs_db se puede usar para iniciar dicha reparación en un sistema de archivos XFS. El comando xfs_db se utiliza para realizar opciones de depuración y posibles reparaciones en un sistema de archivos XFS, pero si se usa de forma inexperta puede hacer que el sistema de archivos sea inutilizable e irrecuperable.

Para usar xfs_db en modo experto, use el comando:

Importante

¡El modo experto del comando xfs_db permite la modificación de estructuras de datos, lo cual puede ser peligroso!

root@localhost:~# xfs_db -x /dev/sda3

Como método para mantener sus sistemas de archivos XFS en buen estado, el comando xfs_fsr o "Reorganizador del sistema de archivos para XFS" se puede usar en sus sistemas de archivos XFS montados para mantenerlos organizados y funcionando bien.

El comando xfs_fsr funciona en un archivo a la vez, anotando dónde están sus datos, compactando y reorganizando los archivos para que sean lo más optimizados posible con el fin de mejorar los tiempos de lectura y escritura, lo que afecta el rendimiento del sistema.

Antes de invocar el comando xfs_fsr para usar los recursos del sistema reorganizando los sistemas de archivos, es una buena idea verificar si necesitan este tipo de mantenimiento. El comando xfs_db se puede usar para evaluar la necesidad de reorganización:

root@localhost:~# xfs_db -r /dev/sda3
xfs_db>frag
actual 345, ideal 289, factor de fragmentación 16.23%
Nota, este número es en gran parte insignificante.
Los archivos en este sistema de archivos promedian 1.19 extensiones por archivo
xfs_db>quit

El uso del comando interno frag informará el estado de fragmentación del sistema de archivos, y esto funcionará en sistemas de archivos montados o desmontados. El estado del sistema de archivos de ejemplo tiene una fragmentación bastante baja, por lo que la reorganización probablemente produciría pocos o ningún resultado real en el rendimiento.

Si el factor de fragmentación supera el 25% aproximadamente, entonces la reorganización produciría resultados de rendimiento medibles.

Para reorganizar y optimizar todos los sistemas de archivos montados en su sistema y ver una salida detallada de lo que se está haciendo exactamente con qué archivos, use el comando:

root@localhost:~# xfs_fsr -v /dev/sda3
No se realizará ninguna mejora (saltando): ino=3147588472
ino=1247877226
No se realizará ninguna mejora (saltando): ino=1247877226
ino=124787727
No se realizará ninguna mejora (saltando): ino=1247877440
ino=1247877470
extensiones antes:2 después:1 HECHO ino=1247877470
ino=1247849901
extensiones antes:2 después:1 HECHO ino=1247849901
ino=1247849973
No se realizará ninguna mejora (saltando): ino=1247849973
ino=1247877640
Salida Omitida...

Otra característica muy útil del comando xfs_fsr es que puede configurarlo para que se ejecute durante un período de tiempo determinado, por lo que se puede iniciar manual o programadamente para que se ejecute durante un par de horas cada noche cuando el sistema está menos utilizado. Para ejecutar xfs_fsr durante 1 hora y luego detenerlo:

root@localhost:~# xfs_fsr -t 3600

La utilidad entonces trabajará hasta por una hora, reorganizando los archivos de las unidades archivo por archivo hasta que se acabe el tiempo límite o los archivos terminen de reorganizarse.

Si por alguna razón xfs_fsr se interrumpe o necesita ser detenido, mantiene un registro de dónde se encontraba en el proceso de reorganización y puede reanudar donde lo dejó cuando se ejecute de nuevo.
